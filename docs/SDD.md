# tscode Software Design Document

## 1. Architecture Overview

`tscode` is a Rust terminal application built on:

- `ratatui` for layout and rendering
- `crossterm` for terminal events, raw mode, alternate screen, and mouse capture
- `syntect` for syntax highlighting
- `portable-pty` for a real cross-platform shell-backed pseudo terminal
- `vt100` for terminal screen parsing and scrollback

The application is a single event loop that updates an in-memory model and redraws the full screen after meaningful input.

## 2. Module Layout

```text
src/
  app.rs        application state and actions
  fs_tree.rs    filesystem tree loading, flattening, and selection
  main.rs       terminal setup, event loop, panic restoration
  shell.rs      PTY shell session, terminal parser, input/output routing
  syntax.rs     syntect loading and line highlighting
  ui.rs         ratatui layout and widgets
```

## 3. State Model

### App

The top-level state owns:

- workspace root
- file explorer state
- opened tabs
- active tab index
- focused panel
- hover target
- quick panel state for quick open, code suggestions, language-server hover, workspace search, document symbols, workspace symbols, go-to-definition candidates, references, problems, source control, run task, and command palette
- active completion state that records the editor path, line, replacement range, and identifier prefix used by the current suggestions panel, plus an app-level cache of language-server completion candidates for that panel
- cached workspace problem diagnostics collected from external project checkers
- line-level problem summaries derived from cached diagnostics for editor gutter/status rendering
- explorer visibility state for dotfile visibility, generated-folder visibility, and visible-tree filtering
- cached Git status and dirty-parent-directory markers for explorer badges
- terminal layout state for normal height and maximized mode
- integrated terminal sessions and the active terminal index
- terminal state
- cached UI hit regions from the most recent draw
- syntax highlighter

Startup canonicalizes the requested CLI path before building this state. Directory paths become the workspace root. File paths are split into parent-directory workspace root plus an initial editor file; the initial file is opened, revealed in the explorer, and the first integrated terminal is spawned in that parent directory. `--help` and `--version` are handled in `main.rs` before raw mode or alternate-screen setup so package and installer checks can verify the binary without needing an interactive terminal.

### Explorer

The explorer stores a tree of `FsNode` values. Directories are loaded lazily when expanded. Filesystem metadata such as file size, modified time, and read-only state is captured when entries are loaded. The tree owns an `ExplorerSortMode` that sorts loaded children with folders first and then by name, type, modified time, or size. Changing sort mode resorts already-loaded children in place, and app-level sort commands remember the selected path before resorting so selection remains on the same visible file or folder whenever possible. Refresh and reveal pass the same sort mode into newly loaded children, so external filesystem changes and lazy expansion do not silently fall back to name order. A flattened visible row list is produced after changes and during rendering, then filtered by app-level visibility state so hidden, ignored, and generated entries can stay hidden by default without changing the underlying tree. `App` maintains a cached set of workspace-visible paths produced by `ignore::WalkBuilder`; the walker respects `.gitignore`, `.ignore`, parent/global Git ignore files, and built-in generated folders when ignored entries are hidden. Quick open, workspace search, workspace symbols, references, definition lookup, rename-symbol, and replace-in-files use the same walker so explorer visibility and workspace scans agree. Explorer reveal expands path ancestors and selects the requested file or folder row. New File and New Folder prompts derive their initial workspace-relative prefix from the selected folder or selected file's parent. The prompt resolver treats simple names as children of that selected base, explicit relative paths as workspace-root-relative targets, rejects parent-directory traversal and workspace-escaping absolute paths, opens newly created files, and reveals newly created folders. Explorer rename accepts one simple file or folder name, rejects workspace-root rename and existing targets, then remaps open tabs and navigation history after the filesystem move. Explorer multi-selection is stored on `App` as a `BTreeSet<PathBuf>` plus an optional anchor path for range selection. `Space` toggles the focused row, modifier-click toggles rows without opening them, and shift-click replaces the selection with the visible range between the anchor and clicked row. Rendering gives multi-selected rows a distinct background and a `*` marker, while keeping the active cursor row and hover highlight separate. Refresh, sorting, filtering, visibility toggles, and folder collapse prune selected paths that are no longer visible, so hidden rows cannot be moved or deleted accidentally. Right-clicking an explorer row updates the selected row and opens `QuickPanelKind::ExplorerContextMenu`, which is populated from the current node, explorer clipboard state, active multi-selection, and active sort mode. Each context menu item carries a `CommandAction`, so mouse activation reuses the same real file, path-copy, terminal-here, sort, refresh, collapse, and visibility functions as keyboard shortcuts and the command palette. Editor and terminal context menus reuse the same `ContextMenuAction` to `QuickItem` conversion path: editor right-clicks can move the cursor before opening `QuickPanelKind::EditorContextMenu`, and terminal right-clicks open `QuickPanelKind::TerminalContextMenu` only when the active child has not requested xterm mouse reporting. When an explorer filter is applied, the app also scans real workspace paths under the same visibility rules and reveals matching files or folders, which expands collapsed ancestors so nested matches are visible even before the user manually opens each parent directory. The app keeps a lightweight workspace snapshot made from visible path, directory/file kind, length, modified time, and read-only metadata. The main loop periodically compares that snapshot while draining terminal output; if files are created, deleted, renamed, or metadata changes outside app-owned explorer commands, the explorer performs a silent refresh that preserves expanded paths, current selection, and sort mode where possible, reapplies active filters, expands newly matching filtered paths, and refreshes Git badges without replacing unrelated user-facing messages.

When the workspace is inside a Git repository, `App` shells out to `git status --porcelain=v1 -z --untracked-files=all` at startup, after explorer refreshes, and after editor saves. The parser maps porcelain records to absolute paths, preserves the index/worktree X/Y state for Source Control commands, and derives dirty parent-directory markers so the renderer can show file-level badges such as `git:M` or `git:?` and folder-level `git:*` badges without mixing Git state into the filesystem tree nodes. Source Control also shells out to `git diff --unified=0 --no-ext-diff --no-color --find-renames HEAD --`, with a fallback for repositories without `HEAD`, then parses diff file headers and hunk ranges into clickable source locations. Stage rows call `git add` for a selected path or `git add -A` for all changes; unstage rows call `git restore --staged` with reset/rm-cached fallbacks so older repositories and Git versions can still remove paths from the index.

Explorer clipboard state stores copy/cut intent and one or more source paths. File-operation commands derive their target list from the active multi-selection when it is non-empty, otherwise from the focused explorer row. Before batch copy, cut, duplicate, or delete, the app normalizes selected paths by removing duplicates and descendants of already-selected parent folders. Paste performs real filesystem copy or move operations, recursively copies directories, creates non-conflicting copy names, and updates open editor tab paths after moves. Explorer delete closes clean open tabs under deleted paths, prunes navigation history for those paths, and refuses the entire filesystem delete when any matching file-backed tab is dirty so unsaved editor contents cannot be discarded by a tree operation.

### Editor

Each opened file tab stores:

- absolute path
- display name
- decoded lines
- vertical scroll offset
- horizontal scroll offset
- cursor line and column
- optional primary selection anchor
- extra normalized occurrence-selection ranges
- extra cursor positions for continued multi-occurrence typing after a selection replacement
- folded line starts used to hide code-block interiors in the editor viewport
- dirty state
- Untitled/file-backed state
- last-known backing-file metadata and external file status
- trailing-newline state
- bounded undo and redo stacks

Editor buffers support insertion, deletion, newline, paste, cursor movement, word movement, selection, mouse multi-cursors, occurrence selection, save, undo, redo, in-file search, go-to-line, symbol navigation, go-to-definition, find-references, code actions, rename-symbol, code suggestions, signature-help, code folding, format-document, diagnostics display, external backing-file detection, and line commands. Untitled tabs are ordinary editor buffers with an `untitled` flag, a display title such as `Untitled-1`, no filesystem placeholder, and no backing-file change checks until Save As retargets them to a real path. Opened file paths are canonicalized before tab lookup so symlinked macOS paths such as `/tmp` and `/private/tmp` do not create duplicate tabs or break relative-path features. Selection is stored as an anchor plus the current cursor position, with additional normalized occurrence-selection ranges for `Ctrl-D` and `Ctrl-Shift-L`. The app keeps an `editor_selection_dragging` flag for left-button drags that start in the editor body; drag and release events keep updating the active selection even if the pointer leaves the body, clamp coordinates to the nearest rendered editor cell, and edge-scroll the viewport before translating the endpoint through the line-number gutter, vertical scroll, folded visible-line map, and horizontal scroll offset. Explicit mouse multi-cursors are stored as extra cursor positions beside the primary cursor; `Alt`+click toggles a cursor after translating terminal coordinates through the editor body, line-number gutter, and horizontal scroll offset. Code folding is stored as a view-only set of folded start lines on `EditorTab`; fold ranges are detected with delimiter matching for braces/brackets/parentheses and an indentation fallback. `visible_line_indices` maps screen rows back to real file line numbers for rendering, vertical scrolling, and mouse coordinate translation, while cursor visibility unfolds a containing block if a navigation command moves the cursor into hidden lines. Fold state is cleared on text edits/reloads so it never dirties the buffer or points at stale content. The same normalized range model is used when copying, cutting, deleting, rendering, replacing ranges, or deriving selected line ranges for block commands. Occurrence selection starts from the active single-line selection or the identifier under the cursor; identifier mode uses whole-word boundaries so `alpha` does not match `alphabet`. Copy and cut join selected ranges in document order, and typing, paste, `Enter`, `Backspace`, or `Delete` applies one bottom-up replacement pass to every selected range or active cursor set under a single undo snapshot. The same selection normalization feeds code navigation and rename: if the selection is a single identifier it is used as the lookup token, otherwise the token under the cursor is derived from the active line. App-level editor navigation history stores bounded back/forward stacks of canonical file, line, and column locations before quick-panel result jumps, definition jumps, go-to-line jumps, and terminal file-reference jumps; `Alt-Left`, `Alt-Right`, and command palette actions pop those stacks without creating a new history entry, and explorer rename/move/delete operations remap or prune stored paths while leaving Untitled tabs untouched. Rename Symbol first attempts an installed language-server `textDocument/rename` request and applies returned `WorkspaceEdit` text edits; open tabs become undoable dirty buffers and closed files are written after range validation. If no valid LSP edit is available, it falls back to an app-layer workspace edit that validates the new identifier, scans visible workspace text files with identifier-boundary matching, applies replacements to open tabs through `EditorTab::replace_entire_text_as_edit`, and writes replacements for closed files directly to disk while skipping binary, oversized, hidden, or generated paths according to visibility settings. Trigger Suggest is an app-layer editor command: it records a `CompletionState` for the active file, current line, identifier replacement range, and prefix before the cursor, requests one batch of installed-language-server completions when available, then opens a quick panel populated from cached LSP candidates, lightweight workspace symbols, identifier tokens from the shared visible text-file provider, dirty open buffers, and file-type keywords. Activating a suggestion revalidates that the target tab is still active and uses `EditorTab::replace_range_as_edit` so the completion is undoable, dirty only when text changes, and never writes to disk until save. Signature Help is an app-layer LSP command that uses the active cursor `DocumentPosition`, requests `textDocument/signatureHelp`, and opens a read-only quick panel containing signatures, active signature/parameter metadata, and documentation preview rows. Workspace and language-server diagnostics are cached as quick-panel items, then grouped by active file and line into `LineProblemSummary` values. The renderer uses those summaries for error/warning/info gutter markers, subtle line backgrounds, active-file problem counts, and active-line status messages while leaving the editor buffer untouched. Copy and cut update the internal editor clipboard and queue an OSC52 terminal clipboard export when the selection is small enough for terminal-safe transmission. Smart editing lives inside `EditorTab`: printable pair-open characters insert matching closing pairs, selected text can be wrapped by pair characters, existing closing pairs can be skipped over, paired Backspace deletes both sides, and newline insertion preserves or increases indentation depending on surrounding code. Long-line navigation tracks a horizontal scroll offset derived from the editor body width after the line-number gutter is removed; rendering crops styled spans after syntax/search/cursor/selection/styling so horizontal scrolling does not discard visual metadata. Line commands include indent, outdent, duplicate, delete, move up/down, and file-type-aware line-comment toggling for either the current line or the selected line range, with each command captured as one undo snapshot. The trim-trailing-whitespace command also lives at the editor-buffer layer so it mutates real lines, preserves trailing-newline state, and participates in undo/redo before any later save writes the cleaned text to disk. Each file-backed `EditorTab` records a lightweight file stamp from metadata at open, save, reload, Save As, and app-managed rename time. The main event loop compares open-tab stamps while draining terminal output: clean modified files are reread with lossy UTF-8 decoding and installed through `set_clean_text`, dirty modified or deleted files keep their in-memory edits and expose `modified on disk` or `deleted on disk`, and Save File/Save All refuse to overwrite non-clean backing files until Reload/Revert or Save As is used. Save File on an Untitled tab opens Save As; Save All skips dirty Untitled tabs and reports that Save As is required. Save As is an app-layer command that prompts for a relative workspace path or absolute path, creates missing parent directories, writes the current in-memory buffer, retargets the active tab path/title, clears dirty state, refreshes explorer and Git status, reveals workspace targets, and refuses a target that is already open in another dirty tab. Format Document is an app-layer command that first asks an installed language server for `textDocument/formatting`, applies returned `TextEdit` values through the same LSP edit path used by rename and code actions, and falls back to piping the current in-memory buffer through an external formatter when no LSP formatting edit is available. Revert File clears an Untitled buffer or rereads a file-backed tab from disk; Reload File From Disk uses `EditorTab::set_clean_text` to replace the buffer, clear selection/history, reset dirty and external-file state, and keep scrolling/cursor state clamped to the reloaded content. The first prerelease includes targeted one-shot LSP hover, signature-help, definition, references, semantic rename, completion, document symbols, workspace symbols, formatting, workspace-edit and command-backed code actions, and diagnostics support, while full VS Code language-service parity, long-lived diagnostics streaming, and project-wide server state remain future work.

Dirty tab close confirmation is handled at the app layer instead of inside `EditorTab`: `close_tab` closes clean tabs directly, but dirty tabs open `QuickPanelKind::DirtyClose` with Save and Close, Don't Save, and Cancel rows. The rows dispatch through normal `CommandAction` variants, so keyboard activation, mouse hover/click activation, tab close targets, middle-click, context menus, and the command palette share one path. Save and Close writes file-backed tabs before removing them; Untitled tabs switch to `PromptKind::SaveAsClose`, reuse the Save As writer, then close the retargeted clean tab after the new file is created.

Editor mouse coordinate translation for cursor placement, drag selection, and `Alt`+click multi-cursors applies the editor body origin, line-number gutter, vertical scroll, folded visible-line map, and horizontal scroll offset before touching buffer positions.

Editor symbol hover is stored as transient app state separate from the editor cursor. Mouse movement over the editor body translates terminal coordinates through the editor body, gutter width, vertical visible-line mapping, and horizontal scroll offset to find the hovered buffer location. If that location is inside an identifier, the app records an `EditorHoverInfo` containing the symbol, hovered file location, definition count, reference count, and the first matching workspace definition/preview. The calculation reuses the same visible workspace text-file provider as Go To Definition and Find References, so dirty open buffers are read from memory and hidden/generated path visibility follows the current explorer settings. The renderer displays the hover as a small overlay near the mouse and mirrors the symbol summary in the status bar; moving outside the editor clears it without changing selection, cursor, or history.

Language-server support is isolated in `src/lsp.rs`. The app builds a `DocumentPosition` from the active in-memory editor buffer, then the LSP module starts a supported stdio server such as `rust-analyzer`, `pyright-langserver --stdio`, `typescript-language-server --stdio`, `gopls`, or `clangd`, performs `initialize` and `initialized`, publishes the buffer with `textDocument/didOpen`, sends one request or waits for `textDocument/publishDiagnostics` notifications, and shuts the server down. The request path understands `Content-Length` JSON-RPC framing, file URI conversion, UTF-16 LSP columns, hover content shapes, signature-help signatures with active signature/parameter indexes and string or UTF-16 offset parameter labels, `Location`/`LocationLink` definitions, Call Hierarchy prepare items and incoming/outgoing call responses, hierarchical `DocumentSymbol` and flat `SymbolInformation` document-symbol responses, `workspace/symbol` responses using either full `Location` ranges or URI-only workspace-symbol locations, reference locations, `WorkspaceEdit` text edits for rename and code actions, `TextEdit` arrays from `textDocument/formatting`, `workspace/executeCommand` for edit-producing code-action commands, server-initiated `workspace/applyEdit` requests, array or list-style completion responses, code action objects/command-only objects, and diagnostic severity/source/code/message fields. `TSCODE_LSP_COMMAND`, `TSCODE_LSP_LANGUAGE_ID`, and `TSCODE_LSP_NAME` can override the server for testing or custom languages. Spawn failures, protocol failures, and timeouts return empty results so the editor can keep using workspace-scan suggestions, definitions, references, rename fallback, document-symbol fallback, workspace-symbol fallback, external formatter fallback, or existing Problems state. This is intentionally one-shot rather than a long-lived language-server session; long-lived diagnostics streaming and project-wide server state remain future work.

Go to Type Definition and Go to Implementation reuse the same one-shot `DocumentPosition` request path and `Location`/`LocationLink` parser as Go to Definition, but call `textDocument/typeDefinition` and `textDocument/implementation`. The app opens a single returned location immediately or stores multiple returned locations in dedicated quick-panel modes so filtering and mouse activation work like other navigation result lists.

Show Incoming Calls and Show Outgoing Calls use a multi-request one-shot LSP session: after `textDocument/prepareCallHierarchy`, the app sends `callHierarchy/incomingCalls` or `callHierarchy/outgoingCalls` with the exact prepared item object returned by the server so opaque `data` fields survive the follow-up request. Incoming call rows prefer the first returned `fromRanges` call site in the caller file; outgoing call rows jump to the callee item's `selectionRange`. Both result sets are stored as filterable quick-panel rows with server name, symbol kind, optional detail text, call-site count, and a source-line preview.

### Quick Panel

The quick panel stores a mode, query text, result list, selected index, and scroll offset. Quick Open recursively scans workspace files while applying the same hidden/generated visibility policy as the explorer, then fuzzy matches path fragments. Suggestions reuse the quick-panel overlay but activate through the editor edit path instead of opening files: the query starts with the cursor prefix, filtering uses the same fuzzy scorer as file and command pickers, and selection replaces the stored completion range. Code Actions also reuse the overlay, but each row points back to a cached LSP `LspCodeAction`; activation applies `WorkspaceEdit` text edits through the same open-buffer and closed-file mutation path as LSP rename, while command-only rows execute `workspace/executeCommand` and collect any server-initiated `workspace/applyEdit` edits before applying them through that same path. Dirty Close uses the same overlay model for unsaved-tab decisions, with command-backed rows instead of ad hoc prompts. Workspace Search scans bounded-size visible workspace text files through the shared workspace text-file provider, uses in-memory text for dirty open tabs so unsaved edits are searchable before save, marks dirty-buffer result details as unsaved, builds file/line preview results, and opens the selected result at its matching cursor location. Document Symbols asks an installed language server for active-buffer symbols when available, caches that batch for filtering inside `QuickPanelKind::DocumentSymbols`, and falls back to lightweight active-buffer extraction when no LSP symbols are returned. Workspace Symbols asks the installed language server for `workspace/symbol` with the current quick-panel query, reuses the current query response while the panel redraws, and falls back to bounded visible workspace text-file heuristics, using in-memory text for dirty open tabs so unsaved declarations are searchable before save. Go To Definition reuses the workspace symbol scan with an exact identifier match first and a case-insensitive fallback, jumping directly when there is one result or showing a definitions panel when there are multiple candidates. Find References scans the same bounded visible workspace text-file set for whole-word occurrences and returns file/line previews whose activation opens the occurrence column. Problems displays cached diagnostics collected by Run Workspace Check or Run LSP Diagnostics, filters them with the quick-panel query, opens the selected diagnostic file at its recorded line and column, and feeds editor line summaries for the active file. Source Control displays Git changed files, command rows for staging or unstaging selected paths and all changes, plus parsed diff hunks, filters them with the same query model, refreshes itself after each index-changing command, and opens an existing hunk path at the new-side hunk line. Run Task detects tasks from `.vscode/tasks.json`, `package.json`, `Cargo.toml`, `Makefile`, `go.mod`, and Python metadata, filters them with the same query model, and activates the selected task through the terminal task runner instead of opening a file. Opening any quick-panel file result records the previous editor location in app navigation history when the destination differs. Replace in Files, Rename Symbol, and Save As use prompt state rather than a result overlay: Replace in Files collects a find string and replacement string, scans the same bounded workspace text-file set, writes replacements to disk, refreshes Git status, updates any clean open tab whose backing file changed, and skips dirty open tabs to avoid overwriting unsaved edits; Rename Symbol collects a new identifier and applies identifier-boundary replacements to open buffers plus closed workspace files; Save As collects the new target path for the active buffer and then dispatches through the app action layer. New Untitled File is a command-palette action that creates an app-owned editor tab directly and does not scan or create workspace files. Run Workspace Check is an app-layer command that detects Cargo, Go, or Python roots, executes the checker with the workspace root as current directory, parses `file:line:column` diagnostics plus Python compile errors into quick-panel items, and reports whether dirty buffers were excluded. Run LSP Diagnostics is an app-layer command that reuses the active in-memory buffer, collects language-server diagnostics, converts them into Problems quick-panel rows, and opens the panel. Command Palette uses the same overlay model with `CommandAction` entries instead of file paths; activating a command dispatches through the app action layer.

LSP hover uses the quick panel as a read-only detail surface. LSP Signature Help uses `QuickPanelKind::SignatureHelp` as another read-only detail surface: rows keep the active file location for context, show the server, active signature, active parameter label, and preview signature or parameter documentation, and close back to editor focus when activated. LSP completions are fetched once before opening `QuickPanelKind::Completions` and then mixed into the same ranked candidate map as workspace symbols and keywords. LSP document symbols are fetched once before opening `QuickPanelKind::DocumentSymbols`, flatten hierarchical children with container context, and fall back to local extraction when empty or unavailable. LSP workspace symbols are attempted by `QuickPanelKind::WorkspaceSymbols` for each new query and fall back to bounded visible workspace text-file heuristics. LSP definitions are attempted before the workspace definition scan; multiple LSP locations reuse `QuickPanelKind::Definitions`, and empty or failed responses fall through to the existing workspace search path. LSP Call Hierarchy result batches use `QuickPanelKind::IncomingCalls` and `QuickPanelKind::OutgoingCalls` so callers and callees can be filtered and opened like other semantic navigation result lists. LSP Document Highlight is an editor-state overlay instead of a quick panel: the command requests `textDocument/documentHighlight`, stores line and character ranges on the active `EditorTab`, keeps read/write/text kinds from the server, and lets rendering overlay those ranges without dirtying the buffer. LSP references are attempted before the workspace references scan and use `QuickPanelKind::LspReferences` so returned server locations can be filtered without being replaced by local fallback results. LSP rename is attempted before workspace text rename; returned edits are applied through the same open-buffer and closed-file mutation paths used by local workspace edits. LSP formatting is attempted before external formatter fallback and applies returned `TextEdit` values through the same range-validated edit path. LSP code actions request active-buffer diagnostics first when available, send them in the code-action context, list returned actions through `QuickPanelKind::CodeActions`, apply action `WorkspaceEdit` values through the same range-validated edit path, and execute command-only actions when the server responds with `workspace/applyEdit` edits. LSP diagnostics are collected from `publishDiagnostics` notifications and reused by the same Problems panel and editor-gutter rendering as workspace checker diagnostics.

LSP type definitions, implementations, and Call Hierarchy are attempted only through installed language servers because there is no reliable syntax-only fallback for those semantic relationships. Empty, failed, or timed-out responses leave the editor in place and report a status message instead of substituting a misleading text scan.

### Terminal

Each integrated terminal session owns:

- a native PTY master/slave pair
- a spawned platform shell running in the workspace root
- a background reader thread
- a writer for user input
- a `vt100::Parser` screen with scrollback

The app stores a vector of terminal sessions, an active terminal index, and a monotonic id used for stable terminal tab titles. Keyboard input while terminal-focused is encoded as terminal byte sequences and written to the active PTY, except for app-owned terminal shortcuts such as terminal search and terminal clipboard actions. When the active child has entered alternate-screen mode or requested mouse reporting, the child is treated as owning terminal app keys: terminal search shortcuts, terminal tab management shortcuts, maximize shortcuts, and scrollback shortcuts are forwarded to the PTY instead of being intercepted. `F6` and ``Ctrl-` `` remain app-level escape hatches for moving focus out of the terminal, and `Ctrl-Shift-C`/`Ctrl-Shift-V` remain host-style terminal copy/paste actions. PTY output is drained from every open terminal session so background terminals keep their screen state up to date. `ShellPanel` also scans the raw byte stream for OSC 7 `file://` current-directory reports and OSC 0/2 terminal-title reports before passing bytes to `vt100`; zsh sessions get a temporary `ZDOTDIR` startup hook and bash sessions get a `PROMPT_COMMAND` hook so ordinary `cd` commands report cwd changes automatically, while other shells can still update cwd if they emit OSC 7 themselves. The app consumes parsed cwd reports during terminal drain and updates `TerminalSession.cwd`, which feeds the terminal header, context-menu details, New Terminal Here behavior, and restart-terminal spawn directory. Unlocked terminal sessions also consume parsed OSC 0/2 title reports to update the tab/header title, while Rename Terminal sets a title lock so user-chosen names are not overwritten by later child output. `ShellPanel` polls the child with nonblocking `try_wait`; when the PTY child exits, `TerminalSession` stores an `exit:N` or `signal:NAME` status so the tab and panel header can show why the shell stopped until restart or close. The terminal renderer reads parser state for scrollback offset, alternate screen, bracketed paste, mouse reporting, cursor visibility, search match count, and cwd/live-or-exit-status session state so the terminal header reflects what the child shell or terminal application is doing.

Terminal management commands reset only the active session's `vt100::Parser` for clear-terminal, kill and replace only the active session for restart-terminal, prompt for, update, and lock only the active session title for rename-terminal, create new independent PTY sessions for new-terminal, create side-by-side independent PTY panes for split-terminal, and kill/remove terminal sessions for close-terminal. Restart-terminal clears any stored exit status for that session. Closing the last terminal restarts it instead of leaving the app without a shell.

Each terminal session also stores the working directory used to spawn its PTY and the latest tracked child-shell directory. Normal new-terminal uses the workspace root, while New Terminal Here derives the selected explorer folder or selected file's parent folder and stores that path on the session. Split Terminal uses the active terminal session's current tracked working directory, stores the previous active terminal index as the split partner, creates a fresh PTY session, focuses the new pane, and leaves both shells alive. Run Task creates a new terminal session titled with the selected task label, spawns it in the task working directory, sends the task command through the same submitted-command path used for editor-to-terminal execution, and leaves existing terminals alive. Rename-terminal updates and locks the tab/header title without touching the PTY child, terminal id, working directory, or session count. Restart-terminal reuses the active session's latest tracked working directory so per-folder shell context survives a shell restart.

Terminal Search is an app-owned overlay on top of the real PTY session. `ShellPanel` can snapshot the `vt100` parser's accessible scrollback by temporarily moving the parser scrollback offset across the retained buffer and restoring the user's offset afterward. Search results are stored as stable global row and character-column ranges tied to the terminal session id. Activating a match moves the parser scrollback offset so the matched row is visible, while the renderer maps visible rows back to global rows and splits terminal spans to apply normal and active-match highlight backgrounds without changing the PTY child state.

## 4. Rendering Design

The UI uses a vertical root layout:

1. title bar
2. body
3. status bar

The body uses a horizontal split:

1. file explorer
2. editor column

The editor column normally uses a vertical split:

1. tab strip and code view
2. integrated terminal

Each render pass records clickable and hoverable rectangles into `HitRegions`. Editor rendering derives its visible rows through `EditorTab::visible_line_indices`, displays `-` and `+` fold markers in the gutter for foldable and folded lines, and appends a compact folded-line summary to folded block headers. It overlays LSP document highlights on top of syntax-highlighted spans with distinct text/read/write backgrounds while keeping selection, cursor, and active search visuals higher priority. It also overlays active-file workspace or language-server diagnostics from the cached Problems list: the gutter uses one marker cell for the highest-priority line diagnostic, and the line receives a severity-tinted background underneath syntax, symbol highlights, search, cursor, and selection spans.

When terminal maximized mode is active, the editor column is replaced by the integrated terminal so command output can use the main workspace area while the title, explorer, and status bars remain visible. The terminal panel renders a one-line terminal tab strip above the active PTY screen when height allows. The tab strip records hit regions for terminal activation, terminal close, and new-terminal actions. When split terminal state is active and enough columns are available, the terminal body is divided into two side-by-side PTY panes with a separator. Each pane records its own body hit region and receives an independent `ShellPanel::resize`, while the active pane keeps the cursor and keyboard focus. Mouse clicks, wheel events, terminal selections, child-owned mouse forwarding, and file-reference resolution first resolve the body under the pointer and switch the app's active terminal index to that pane. Closing a terminal adjusts or clears the stored split partner index so stale pane state cannot point at a removed session. In the normal split layout, the terminal top border is also recorded as a resize hit region; left-dragging that border updates `App::terminal_rows` with minimum and editor-preserving maximum clamps, focuses the terminal, and highlights the border while hovered or dragging.

## 5. Input Design

### Mouse Move

Mouse movement updates `HoverTarget` by checking recorded hit regions in front-to-back order.

### Mouse Click

Mouse clicks use the current coordinate to:

- focus explorer/editor/terminal
- toggle a directory
- open a file
- select a tab
- close a tab through its tab-strip close target
- activate a quick-panel result when an overlay is visible
- toggle a code fold when the click lands on an editor gutter fold marker
- toggle an additional editor cursor with `Alt`+click

### Mouse Wheel

Vertical wheel events route to the hovered panel if known, otherwise to the focused panel. Horizontal wheel events over the editor pan the active tab's long-line viewport.

### Keyboard

Keyboard events map to panel-specific actions. Quick-panel input handles query editing, result movement, and activation before normal panel shortcuts. `F1` opens the command palette because many terminal sessions cannot reliably distinguish `Ctrl-P` from `Ctrl-Shift-P`. Explorer-focused input supports tree filtering and visibility toggles in addition to file operations. Editor-focused input supports save, search, replace, go-to-line, repeated search, undo, redo, selection, occurrence selection, word movement, internal clipboard operations, terminal clipboard export, symbol navigation, go-to-definition, find-references, code-action, trigger-suggest, code folding, format-document, line commands, and saved-tab close shortcuts. Context menus are QuickPanel instances with command-backed items, so mouse activation and keyboard activation converge on `run_command`. Workspace-level shortcuts open quick search, replace prompts, or the Run Task picker before panel-specific handling when terminal focus is not active. Terminal focus escape shortcuts are handled before PTY forwarding so `F6` and ``Ctrl-` `` can move out of the integrated terminal even while a full-screen terminal child is focused. Normal shell-mode terminal focus also supports app-owned terminal search, tab management, maximize, and scrollback shortcuts. Once the parser reports alternate screen or terminal mouse reporting, those convenience shortcuts and terminal-body right-clicks are considered child-owned and are sent through the PTY. Terminal `Ctrl-Shift-C` and `Ctrl-Shift-V` are intercepted for terminal selection copy and clipboard paste, while plain `Ctrl-C` and other terminal-focused input are forwarded to the PTY. The app-level exit shortcut is `Ctrl-Q` so `Ctrl-C` can be delivered to the shell when terminal focus is active. Dirty editor buffers trigger an explicit quit confirmation instead of exiting immediately.

### Editor Clipboard

The editor clipboard is an in-memory application clipboard used by `Ctrl-C`, `Ctrl-X`, and `Ctrl-V` in editor focus. It is deliberately separate from explorer copy/cut state and from terminal `Ctrl-C`, which remains a PTY signal when terminal focus is active. Range deletion and replacement are performed as single undoable edits, including multi-occurrence selections.

For host/terminal clipboard integration, `App` stores one pending clipboard export string. The main event loop takes that value after rendering, encodes it as an OSC52 payload, and writes it directly to the terminal backend. This avoids GUI clipboard dependencies that would weaken SSH behavior or cross-platform release builds. Editor copy/cut, terminal selection copy, and command-palette path-copy actions share this export path; explorer file copy/cut remains a filesystem operation clipboard.

In-file search state lives on `App::search_needle`. The editor renderer overlays visible search-match highlights on top of plain line text when a match is present, while keeping explicit editor selections higher priority. The status bar computes the active file's match count from the same search string.

Replace in file is modeled as a two-step prompt: find text, then replacement text. Single replace changes the current match when the cursor is on one or otherwise searches forward with wraparound. Replace-all updates every active-buffer line match in one `EditorTab` undo snapshot so a single undo restores the previous file contents.

Run Selection in Terminal is an editor-to-PTY bridge in the app action layer. The app derives the active selection, falling back to the current non-blank line, normalizes CRLF/CR line endings to LF, appends a final enter when needed, converts enters to carriage returns for the PTY, writes the bytes through `ShellPanel::send_text`, and focuses the integrated terminal so the submitted command and output are immediately visible.

Run Task is a workspace-to-PTY bridge in the app action layer. Task detection reads real workspace files without executing discovery commands: VS Code JSONC task files are comment-stripped then parsed as JSON, package scripts are read from `package.json`, Cargo/Go/Python default tasks are inferred from project metadata files, and Make targets are parsed from simple target declarations. Activating a task creates a new PTY terminal session for that task and submits the command as shell input so output, ANSI styling, scrollback, terminal selection, and clickable file references all use the same terminal renderer as manually typed commands.

## 6. Syntax Highlighting

`syntect` loads default syntax and theme sets once during startup. The renderer chooses syntax by token or file extension, highlights visible lines only, and converts style foreground colors into ratatui `Color::Rgb`.

If highlighting fails, rendering falls back to plain text.

## 7. Cross-Platform Shell Design

The integrated terminal uses:

- Unix: `$SHELL` or `/bin/sh`
- Windows: `%COMSPEC%` or `cmd.exe`

The shell runs inside a PTY with `TERM=xterm-256color`, receives resize notifications from the terminal panel, and inherits the workspace root as the current directory.

PTY output is parsed into `vt100` cells. `ShellPanel::styled_rows` groups adjacent cells with the same foreground color, background color, and text modifiers into lightweight terminal spans. The UI layer maps those spans into ratatui `Span` values so command output keeps ANSI color and style information without exposing ratatui rendering concerns to the PTY layer.

Keyboard events are converted to xterm-compatible byte sequences before being written to the PTY. The encoder handles control characters, modified navigation keys, function keys, Shift-Tab, null, and the parser's application-cursor mode so alternate-screen tools receive the arrow-key family they requested.

Paste events use normal byte writes unless the parsed terminal screen has bracketed paste mode enabled. In bracketed paste mode, pasted text is wrapped with the standard begin/end paste control sequences before being sent to the PTY.

Terminal mouse handling has two modes. If the child application has requested xterm mouse events, mouse down, release, drag, move, and wheel events over the resolved terminal pane body are encoded using that pane's parser mouse mode/encoding and passed through to the pane's PTY. SGR mouse mode uses numeric `CSI < ... M/m` sequences, default mode uses the classic three-byte coordinate payload, and UTF-8 mode emits the same payload values as UTF-8 encoded codepoints so wider terminal coordinates reach compatible child TUIs correctly. Otherwise, tscode owns terminal-body mouse interaction: a click inspects the clicked visible row and opens an existing file reference in the editor when one is found. The parser first checks the token under the mouse for `path`, `path:line`, and `path:line:column`, then scans the whole row for richer compiler and runtime forms such as quoted paths with trailing line/column suffixes, Python `File "path", line N` tracebacks, and parenthesized `path(line,column)` stack frames. A left-button drag creates a screen-coordinate terminal text selection. The app keeps the selection attached to the active terminal session id, renders selected cell ranges by splitting terminal spans and applying a selection background, extracts visible row text on release, copies it into the internal clipboard, and queues the same OSC52 export used by editor/path copy. `Ctrl-Shift-C` re-copies the active terminal selection, and `Ctrl-Shift-V` sends the internal clipboard to the PTY through the same bracketed-paste-aware path used for external paste events.

## 8. Release and Packaging Design

The repository includes:

- `install.sh`
- `.github/workflows/release.yml`
- `dist/` helper scripts for packaging
- Cargo metadata suitable for `cargo-deb` and `cargo-generate-rpm`

The release workflow builds archive artifacts on macOS, Linux, and Windows runners. Linux package jobs generate `.deb` and `.rpm` packages. The workflow uploads all assets to a GitHub prerelease for tags matching `v*`.

## 9. Risks and Mitigations

- Full terminal emulation is complex. The prerelease uses `vt100` parsing and a real PTY, which supports normal shell interaction, xterm-style keyboard input, and common terminal mouse modes, while deeper terminal edge cases remain ongoing work.
- Cross-compiling every target from one machine may require external linkers. CI uses native runners and cross/zig where practical.
- Terminal mouse support varies by emulator. The app uses crossterm's standard mouse capture and also provides keyboard fallbacks.
